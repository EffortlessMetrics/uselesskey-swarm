# ADR-0011: KID Generation via BLAKE3 Truncation

## Status
Accepted

## Context
JWKs require stable, unique identifiers for keys. The kid must be deterministic from key material and URL-safe for use in HTTP headers and query parameters.

## Decision
Key IDs (kid) are generated by hashing public key material with BLAKE3, truncating to 96 bits (12 bytes), and encoding as base64url without padding.

- Hash: BLAKE3 output truncated to 12 bytes (96 bits)
- Encoding: base64url without padding
- Default prefix length: 12 bytes ("enough to avoid accidental collisions")

## Consequences

**Positive:**
- 96 bits provides sufficient collision resistance for test fixture scenarios
- Base64url encoding produces URL-safe, compact identifiers
- Same key material always produces same kid across runs

**Negative:**
- Truncation reduces collision resistance (acceptable for test fixtures)
- Not cryptographically unique (not needed for test fixtures)

## Alternatives Considered
- **Full 256-bit BLAKE3 output:** Longer than necessary, increases JWKS size
- **SHA-256:** Slower than BLAKE3, no benefit for test fixtures
- **Random UUIDs:** Not deterministic, breaks cacheability
- **Simple counter:** Not derived from key material, collisions possible
