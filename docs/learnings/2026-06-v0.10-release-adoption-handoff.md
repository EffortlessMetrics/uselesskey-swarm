# Release Adoption Work Stops at the Source Boundary

## Trigger

The v0.10 release-adoption lane reached the swarm handoff boundary: policy,
schemas, receipts, public claims, support tiers, command ownership, clean-room
CLI/facade adoption, external CI recipes, and the portable fail-closed receipt
consumer were prepared and proven in `uselesskey-swarm`.

## What Changed

After `source-release-handoff` is done, more swarm polish is no longer the
right work. The next candidate work belongs in `EffortlessMetrics/uselesskey`
after explicit approval, where source versions, changelog, package evidence,
release notes, immutable release records, publication, and tags are owned.

## Evidence

- `docs/handoffs/2026-06-19-v0-10-release-adoption-closure-closeout.md`
  records completed work through `source-release-handoff`, deferred source work,
  release-execution blocking, and proof commands.
- `.uselesskey/goals/archive/2026-06-19-v0-10-release-adoption-closure.toml`
  archives the lane with `release-execution` blocked.
- `docs/release/v0.10-source-handoff.md` preserves the swarm-to-source boundary.
- `docs/release/v0.10-adoption-proof-inventory.md` and
  `docs/release/v0.10-command-ledger.md` map release-facing surfaces to proof.

## Rule to Keep

`uselesskey-swarm` prepares, tests, audits, and hands off. `EffortlessMetrics/uselesskey`
owns the source release candidate, publication, tag, and release record.
