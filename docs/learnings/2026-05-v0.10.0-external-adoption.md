# External Adoption Needs Clean-Project Proof

## Trigger

v0.9.x made `uselesskey` command-backed, reviewable, and easier to approach
from the repo. That still left one important product question: whether a user
could get value from a clean project and installed CLI-shaped workflow without
learning the repo operating system first.

## What Changed

The v0.10.0 external adoption buildout moved the adoption proof outside the
workspace:

```text
clean temp project -> copy dependency or CLI command -> generate bundle
  -> verify or inspect bundle -> understand proof boundary
```

The lane added clean-project external adoption smoke, downstream-shaped
examples, clearer installed CLI and Rust dependency docs, richer CLI
profile/bundle/inspect explanations, and an explicit proof-handoff boundary.

## Evidence

- `cargo xtask external-adoption-smoke --path .` creates clean temporary
  adopter projects and writes Markdown/JSON receipts.
- Published-version mode is available through
  `cargo xtask external-adoption-smoke --version <published>` as an
  audit/reference check, not a release trigger.
- `examples/external/` provides webhook, OIDC/JWKS, TLS, and Rust test fixture
  examples that are exercised by the smoke command.
- User docs now separate installed CLI, Rust dependency, repo-local proof,
  reviewer evidence, and maintainer evidence.
- CLI profile and bundle explanation paths state generated files,
  scanner-safe/runtime-material posture, proof handoffs, and "does not prove"
  boundaries.
- `cargo xtask adoption-regression --external` lets maintainers opt into
  clean-project adoption proof without making the default receipt too broad.

## Rule to Keep

Do not treat repo-local proof as the same thing as adopter proof. Public
examples and dependency snippets should be executable from a clean project, and
the command that proves that path should write a receipt.

Keep `xtask` as the maintainer/reviewer proof plane unless a future installed
proof command has a safe, metadata-only proof engine. The installed CLI should
generate, verify, inspect, and explain fixtures without implying provider
compatibility or production security assurance.
