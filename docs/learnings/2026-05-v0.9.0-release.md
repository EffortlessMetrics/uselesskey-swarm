# A Command-Backed Release Still Needs One Release Authority

## Trigger

v0.9.0 was the first release after the source-of-truth, claim-backed
verification, PR-lite, no-panic new-debt, and webhook contract-pack lanes had
all closed. The release needed to prove the new operating model from candidate
proof through crates.io, docs.rs, and post-release claim receipts.

## What Changed

The release made the fixture-platform claim externally adoptable:

```text
README / crates.io / docs.rs
  -> CHANGELOG v0.9.0
    -> verification docs
      -> public claims
        -> claim-report
          -> claim-proof
            -> verification-pack
              -> release evidence
                -> post-release audit
```

Users can install `0.9.0`, generate TLS/OIDC/webhook contract packs, run stable
claim proof, and build a metadata-only verification pack for review.

## Evidence

- `cargo xtask publish` completed.
- All intended publish crates are visible on crates.io at `0.9.0`.
- `cargo xtask cratesio-smoke --version 0.9.0` passed from the published
  registry view.
- docs.rs returned `{"doc_status":true,"version":"0.9.0"}` for
  `uselesskey 0.9.0`.
- `cargo xtask claim-proof --all-stable` passed.
- `cargo xtask verification-pack --out target/uselesskey-verification` passed.
- TLS, OIDC/JWKS, and webhook bundle proofs passed after publish.

## Rule to Keep

The release candidate proof, cut PR, tag, publish, GitHub release, and
post-release audit are separate boundaries. Do not stop at a green release-prep
or release-candidate PR, and do not treat a queued docs.rs build as a republish
reason.

Keep one authority for GitHub release creation. In v0.9.0, the tag-triggered
Release workflow succeeded through publish but failed creating the GitHub
release because the curated release already existed. That was harmless for the
package release, but it is avoidable automation noise.
