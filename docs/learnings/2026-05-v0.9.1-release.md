# Patch Releases Earn Trust by Publishing Corrected Proof Surfaces

## Trigger

v0.9.0 made `uselesskey` approachable through command-backed claims,
verification packs, first-run UX, and contract-pack proof. After that release,
adoption-confidence work corrected scanner-safe runtime JWK/JWKS metadata and
added a bounded adoption-regression receipt for copied user paths.

The remaining question was whether to leave that confidence work only on `main`
or publish it as a patch users could install from crates.io.

## What Changed

v0.9.1 made the adoption-confidence fixes externally available without
expanding the product promise:

```text
v0.9.0 user path
  -> corrected runtime scanner-safe metadata
    -> adoption-regression receipt
      -> crates.io smoke
        -> post-release audit
```

The release story stayed narrow: no new profiles, contract packs, badges,
provider compatibility promises, or production security claims.

## Evidence

- The tag-triggered Release workflow completed preflight, publish, and GitHub
  release creation successfully.
- All intended publish crates are visible on crates.io at `0.9.1`.
- The removed v0.7.x shim crates were not republished at `0.9.1`.
- `cargo xtask cratesio-smoke --version 0.9.1` passed from the published
  registry view.
- docs.rs returned `{"doc_status":true,"version":"0.9.1"}` for
  `uselesskey 0.9.1`.
- `cargo xtask adoption-regression` passed.
- `cargo xtask claim-proof --all-stable` passed.
- `cargo xtask verification-pack --out target/uselesskey-verification` passed.
- TLS, OIDC/JWKS, and webhook bundle proofs passed after publish.

## Rule to Keep

When a patch fixes public proof metadata, publish it once the release proof is
clean instead of making users infer corrected behavior from `main`.

Keep the patch boundary small. A patch release can improve trust without adding
new fixture families or public claims.
