# v0.10.0 Receipt Contract

This is the released contract freeze for bundle audit packets used by downstream CI and
reviewers.

Use these files as the machine boundary:

- `docs/schemas/bundle-audit.schema.json`
- `docs/reference/bundle-audit-json.md`
- `examples/audit-receipts/`
- `cargo xtask check-audit-receipts`

## Stable receipt fields

For `bundle-audit.json` produced by `uselesskey audit-bundle --out`, downstream
automation should branch on:

- `status`
- `profile`
- `checks[].failure_class`

The following fields are also validated by schema and intended to be stable:

- `version`
- `bundle_path`
- `manifest_path`
- `manifest_version`
- `artifact_count`
- `receipt_count`
- `scanner_safe_count`
- `runtime_material_count`
- `files[]`
- `artifacts[]`
- `receipts[]`
- `missing_files[]`
- `unexpected_files[]`
- `checks[].status`
- `checks[].name`
- `checks[].failure_class`

For `version = 1`, failure classes are fixed to:

`missing_manifest`, `invalid_manifest`, `path_escape`, `missing_artifact`,
`unexpected_artifact`, `missing_receipt`, `invalid_receipt`,
`scanner_safe_mismatch`, `runtime_material_mismatch`, `profile_validation_failed`,
and `unsupported_profile`.

Do not branch on:

- `checks[].detail`
- `boundaries`
- `does_not_prove`
- `bundle-audit.md`

Those are review-facing strings and may change across releases.

## Metadata-only boundary

`bundle-audit.json` is not a payload archive. It must remain metadata-only:

- artifact and receipt metadata only;
- no PEM/JWK/JWKS/JWT/secret values;
- no webhook bodies;
- no runtime key material;
- no local fixture files.

`checks[].failure_class` is the only machine branch key for CI failures.

## Schema compatibility policy

- `version` is currently `1`.
- Breaking contract changes require an explicit schema change in
  `docs/schemas/bundle-audit.schema.json`.
- Compatible schema changes require updating this contract page, related references,
  and the release lane command ledger together.
- New failure classes must include committed `examples/audit-receipts/*.json`
  examples and pass `cargo xtask check-audit-receipts`.

## Allowed proof behavior

- `cargo xtask check-audit-receipts` validates schema coverage for every stable
  failure class example.
- `cargo xtask check-bundle-schemas` validates manifest and audit output for the
  release profiles.
- `cargo test -p uselesskey-cli --all-features audit_bundle` validates CI-path
  behavior.

## Forbidden raw material

If any receipt contains secret-shaped raw payload data, it is out-of-contract and
must fail CI before upload. The failure mode is enforced by:

- `cargo xtask check-audit-receipts` (schema and example coverage)
- `cargo xtask no-blob`
- release command-ledger policy that uploads only `bundle-audit.json`
  and `bundle-audit.md`.
