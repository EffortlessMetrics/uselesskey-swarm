## Decision Rule

Do not advertise `0.10.0` as installable until it is published.

Use these classifications:

| Classification | Meaning | Release-readiness action |
| --- | --- | --- |
| Current stable | Public install or dependency snippet that should continue to say `0.9.1` until v0.10.0 exists on crates.io. | Leave unchanged in readiness PRs. Update only in the version-bump or post-publish docs PR. |
| Release candidate | A readiness-only command, record, or release note may mention v0.10.0 as a target candidate. | Mark as release-candidate wording and keep it out of copyable install snippets. |
| Package version | Workspace crate versions and path dependency version constraints. | Leave unchanged until an explicit version-bump/package dry-run slice. |
| Historical | Prior release evidence, learning, closeout, or audit docs. | Leave unchanged. |
| Internal path dependency | `path = ... version = "0.9.1"` inside workspace manifests. | Leave unchanged until the version-bump slice updates package metadata coherently. |

## Deferred Edits

The version-bump/package slice should decide and update:

- workspace package versions;
- workspace path dependency version constraints;
- public README dependency snippets;
- crate README dependency snippets;
- how-to dependency snippets;
- `cargo install uselesskey-cli --version ...` snippets;
- release notes and readiness records.

That slice must run package and docs proof, not just search-and-replace.

## Boundaries

- No publish.
- No tag.
- No crates.io push.
- No GitHub release.
- No source sync.
- No claim that `0.10.0` is installable before crates.io publication.
