## Release Repo Verification

Before publication from the release/source boundary, verify:

- source sync includes the merged readiness changes;
- package versions and internal path dependency constraints are set for the
  intended release version;
- public snippets are still correct for current stable versus release-candidate
  wording;
- `cargo xtask publish-preflight` passes in the publication repo;
- `cargo xtask publish-check` passes in the publication repo;
- `cargo xtask no-blob` passes in the publication repo;
- installed CLI and facade smokes still pass from the release-candidate source.

## Rollback

If release review rejects this lane:

- revert the specific readiness PR that introduced the rejected evidence;
- return the matching `.uselesskey/goals/active.toml` work item to `ready` or
  `blocked` with the concrete reason;
- do not publish a partial release to compensate for a failed readiness check.

If publication starts and a crate publish fails, use the publish-state resume
flow rather than reordering crates ad hoc.

## Claim Boundary

This lane supports release review for documented test-fixture workflows. It
does not prove:

- production security;
- provider compatibility;
- downstream verifier correctness;
- scanner-policy approval;
- publish success;
- future registry state.
