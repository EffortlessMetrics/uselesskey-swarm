# Test PGP key parsing and policy paths

You are testing code that imports OpenPGP transferable keys, accepts armored
public keys, rejects malformed PGP blocks, or checks key identity through
fingerprints and user IDs. You need realistic PGP-shaped material without
checking generated key blocks into the repository.

The `uselesskey-pgp` crate provides Ed25519, RSA-2048, and RSA-3072 OpenPGP key
fixtures in armored and binary forms. A copyable downstream-style test crate
lives at
[`../../examples/external/pgp-fixture-validation/`](../../examples/external/pgp-fixture-validation/).

## Add the test dependencies

```toml
[dev-dependencies]
pgp = { version = "0.19.0", default-features = false }
uselesskey-core = { version = "0.9.1", default-features = false }
uselesskey-pgp = "0.9.1"
```

Run the repo proof path with:

```bash
cargo xtask external-adoption-smoke --path . --library-examples
```

## Generate an armored PGP fixture

```rust
use std::io::Cursor;

use pgp::composed::{Deserializable, SignedPublicKey, SignedSecretKey};
use pgp::types::KeyDetails;
use uselesskey_core::{Factory, Seed};
use uselesskey_pgp::{PgpFactoryExt, PgpSpec};

let fx = Factory::deterministic(Seed::from_env_value("pgp-test").unwrap());
let key = fx.pgp("release signer", PgpSpec::ed25519());

let (secret, _) = SignedSecretKey::from_armor_single(
    Cursor::new(key.private_key_armored()),
).unwrap();
let (public, _) = SignedPublicKey::from_armor_single(
    Cursor::new(key.public_key_armored()),
).unwrap();

assert_eq!(secret.fingerprint().to_string(), key.fingerprint());
assert_eq!(public.fingerprint().to_string(), key.fingerprint());
assert_eq!(key.user_id(), "release signer <release-signer@uselesskey.test>");
```

Use `PgpSpec::rsa_2048()` or `PgpSpec::rsa_3072()` when your code needs an RSA
OpenPGP shape. The generated key is stable for the same seed, label, and spec.

## Generate a binary PGP fixture

```rust
use std::io::Cursor;

use pgp::composed::{Deserializable, SignedPublicKey, SignedSecretKey};
use pgp::types::KeyDetails;
use uselesskey_core::{Factory, Seed};
use uselesskey_pgp::{PgpFactoryExt, PgpSpec};

let fx = Factory::deterministic(Seed::from_env_value("pgp-binary-test").unwrap());
let key = fx.pgp("binary signer", PgpSpec::ed25519());

let secret = SignedSecretKey::from_bytes(Cursor::new(key.private_key_binary())).unwrap();
let public = SignedPublicKey::from_bytes(Cursor::new(key.public_key_binary())).unwrap();

assert_eq!(secret.fingerprint().to_string(), key.fingerprint());
assert_eq!(public.fingerprint().to_string(), key.fingerprint());
```

## Test failure paths

Generate the negative material inside the test instead of committing payloads:

- **Corrupted armor.** Call `private_key_armored_corrupt(...)` or
  `private_key_armored_corrupt_deterministic("case-id")` and assert your parser
  rejects it.
- **Truncated binary.** Call `private_key_binary_truncated(len)` and assert
  binary import fails.
- **Deterministically corrupted binary.** Call
  `private_key_binary_corrupt_deterministic("case-id")` to drive stable parser
  or checksum rejection.
- **Mismatched public key.** Call `mismatched_public_key_armored()` or
  `mismatched_public_key_binary()`. The public key remains parseable but its
  fingerprint differs from the private key.
- **Unexpected identity.** Generate with a different label and assert your
  policy rejects the user ID or fingerprint.

## What this proves

- Your code can parse the armored and binary OpenPGP key shapes it expects.
- Your code checks fingerprints, user IDs, and public/private key identity.
- Your code rejects corrupted armor, truncated binary material, mismatched
  public keys, or unexpected identities when your test asserts those cases.

## What this does not prove

- Production PGP key custody, rotation, or Web of Trust policy.
- Provider compatibility with every OpenPGP implementation.
- Production signing or encryption security.
- Release readiness, downstream verifier correctness, or production security.

## Scanner-safety boundary

OpenPGP private key blocks are secret-shaped test material even when they are
generated by `uselesskey`. Keep generated armor and binary bytes under
`target/` or in memory, do not copy them into docs or source fixtures, and use
`cargo xtask no-blob` to catch accidental committed payloads.
