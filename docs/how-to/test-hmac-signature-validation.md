# Test HMAC signature validation paths

You are testing code that accepts shared-secret signatures, routes HS256,
HS384, or HS512 policy, and rejects tampered messages, wrong secrets, or wrong
algorithms. You need deterministic secret material without committing generated
bytes to the repository.

The `uselesskey-hmac` crate provides deterministic HMAC secret fixtures with
stable algorithm labels and byte lengths. A copyable downstream-style test
crate lives at
[`../../examples/external/hmac-signature-validation/`](../../examples/external/hmac-signature-validation/).

## Add the test dependencies

```toml
[dev-dependencies]
sha2 = "0.11.0"
uselesskey-core = { version = "0.9.1", default-features = false }
uselesskey-hmac = "0.9.1"
```

Run the repo proof path with:

```bash
cargo xtask external-adoption-smoke --path . --library-examples
```

## Generate a shared-secret fixture

```rust
use uselesskey_core::{Factory, Seed};
use uselesskey_hmac::{HmacFactoryExt, HmacSpec};

let fx = Factory::deterministic(Seed::from_env_value("hmac-test").unwrap());
let secret = fx.hmac("webhook-shared-secret", HmacSpec::hs256());

assert_eq!(secret.label(), "webhook-shared-secret");
assert_eq!(secret.spec().alg_name(), "HS256");
assert_eq!(secret.secret_bytes().len(), 32);
```

Use `HmacSpec::hs384()` or `HmacSpec::hs512()` when your code needs those
algorithm branches. The generated secret is stable for the same seed, label,
and spec.

## Test positive and negative signature policy

Pass `secret.secret_bytes()` into the HMAC, JWT, or webhook library your
verifier already uses. If you only need to prove downstream policy wiring, keep
a tiny test-only helper near the test:

```rust
use sha2::{Digest, Sha256, Sha384, Sha512};
use uselesskey_hmac::{HmacSecret, HmacSpec};

fn verify_fixture_signature(secret: &HmacSecret, message: &[u8], expected: &[u8]) -> bool {
    fixture_signature_for_policy_test(secret, message) == expected
}

fn fixture_signature_for_policy_test(secret: &HmacSecret, message: &[u8]) -> Vec<u8> {
    match secret.spec() {
        HmacSpec::Hs256 => {
            let mut digest = Sha256::new();
            digest.update(secret.spec().alg_name().as_bytes());
            digest.update(secret.secret_bytes());
            digest.update(message);
            digest.finalize().to_vec()
        }
        HmacSpec::Hs384 => {
            let mut digest = Sha384::new();
            digest.update(secret.spec().alg_name().as_bytes());
            digest.update(secret.secret_bytes());
            digest.update(message);
            digest.finalize().to_vec()
        }
        HmacSpec::Hs512 => {
            let mut digest = Sha512::new();
            digest.update(secret.spec().alg_name().as_bytes());
            digest.update(secret.secret_bytes());
            digest.update(message);
            digest.finalize().to_vec()
        }
    }
}
```

Then assert both acceptance and rejection paths:

```rust
use uselesskey_core::{Factory, Seed};
use uselesskey_hmac::{HmacFactoryExt, HmacSpec};

let fx = Factory::deterministic(Seed::from_env_value("hmac-policy-test").unwrap());
let secret = fx.hmac("webhook-shared-secret", HmacSpec::hs256());
let message = br#"{"event":"invoice.paid","id":"evt_test"}"#;
let signature = fixture_signature_for_policy_test(&secret, message);

assert!(verify_fixture_signature(&secret, message, &signature));
assert!(!verify_fixture_signature(
    &secret,
    br#"{"event":"invoice.voided","id":"evt_test"}"#,
    &signature
));

let wrong_secret = fx.hmac("other-webhook-secret", HmacSpec::hs256());
assert!(!verify_fixture_signature(&wrong_secret, message, &signature));

let wrong_algorithm = fx.hmac("webhook-shared-secret", HmacSpec::hs512());
assert!(!verify_fixture_signature(&wrong_algorithm, message, &signature));
```

## What this proves

- Your tests generate deterministic shared-secret material without committed
  secret bytes.
- Your verifier-shaped code can branch on HS256, HS384, or HS512 metadata.
- Your policy rejects tampered messages, wrong secrets, or wrong algorithm
  choices when your test asserts those cases.

## What this does not prove

- Production HMAC implementation correctness.
- Production secret custody, rotation, or storage policy.
- Provider compatibility for webhook or JWT services.
- Webhook or JWT contract-pack completeness.
- Release readiness, downstream verifier correctness, or production security.

## Scanner-safety boundary

HMAC secret bytes are runtime material even when `uselesskey` generated them.
Keep generated bytes in memory or under `target/`, do not copy them into docs
or source fixtures, and use `cargo xtask no-blob` to catch accidental committed
payloads.
