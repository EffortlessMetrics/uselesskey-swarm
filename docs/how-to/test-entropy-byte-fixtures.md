# Test deterministic byte fixture paths

You are testing code that needs byte-shaped inputs but does not need RSA,
ECDSA, Ed25519, PGP, X.509, JWT, or provider-specific semantics. Use
`uselesskey-entropy` for deterministic byte buffers that stay out of source
fixtures and can be regenerated inside tests.

A copyable downstream-style test crate lives at
[`../../examples/external/entropy-byte-fixtures/`](../../examples/external/entropy-byte-fixtures/).

## Add the test dependencies

```toml
[dev-dependencies]
uselesskey-core = { version = "0.9.1", default-features = false }
uselesskey-entropy = "0.9.1"
```

Run the repo proof path with:

```bash
cargo xtask external-adoption-smoke --path . --library-examples
```

## Generate deterministic bytes

```rust
use uselesskey_core::{Factory, Seed};
use uselesskey_entropy::EntropyFactoryExt;

let fx = Factory::deterministic(Seed::from_env_value("entropy-test").unwrap());
let fixture = fx.entropy("api-token-placeholder");

let bytes = fixture.bytes(32);
let same = fixture.bytes(32);

assert_eq!(bytes, same);
assert_eq!(bytes.len(), 32);
assert_eq!(fixture.label(), "api-token-placeholder");
assert_eq!(fixture.variant(), "good");
```

The cache identity includes the label, requested length, and variant. Different
lengths are derived independently, so a shorter buffer is not the prefix of a
longer one:

```rust
let short = fixture.bytes(16);
let long = fixture.bytes(32);

assert_eq!(short.len(), 16);
assert_eq!(long.len(), 32);
assert_ne!(short.as_slice(), &long[..16]);
```

## Test alternate policy cases

Use variants when one test needs stable alternate material, such as a rotated
placeholder, a near-miss value, or a second cache identity:

```rust
let current = fixture.bytes(48);
let rotated = fixture.bytes_with_variant(48, "rotated");

assert_ne!(current, rotated);
```

Use `fill_bytes` when the downstream API expects a mutable buffer:

```rust
let mut buf = [0u8; 48];
fixture.fill_bytes(&mut buf);

assert_eq!(current.as_slice(), &buf);
```

## What this proves

- Your tests can create deterministic byte-shaped inputs without committed
  byte fixtures.
- Your code handles expected buffer lengths and alternate variants.
- Your debug and review surfaces can avoid exposing generated bytes.

## What this does not prove

- Production randomness, key derivation, password generation, or token minting.
- Scanner-policy approval or permission to commit generated bytes.
- Provider compatibility, release readiness, downstream verifier correctness,
  or production security.

## Scanner-safety boundary

Entropy fixtures can still look secret-shaped. Keep generated bytes in memory
or under `target/`, do not paste them into docs or source fixtures, and use
`cargo xtask no-blob` to catch accidental committed payloads.
