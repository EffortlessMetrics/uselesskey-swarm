# USELESSKEY-SPEC-0003: Contract-Pack Profile Requirements

## Problem

Contract packs are one of the clearest ways `uselesskey` turns secret-shaped
fixture generation into reusable test infrastructure. A pack such as TLS or
OIDC/JWKS is valuable only when users can tell:

- which fixtures are generated;
- which positive and negative paths are covered;
- what the pack proves;
- what it explicitly does not prove;
- which command verifies the pack;
- which user guide teaches the test workflow;
- which release lane carries the proof.

Without a reusable contract-pack profile requirement, each new pack can become
a bespoke implementation with inconsistent proof, docs, and claim boundaries.

## Behavior

A contract pack is a named fixture profile with a proof-backed user contract.
It is not just a generated directory.

Every stable contract pack must define:

| Requirement | Meaning |
| --- | --- |
| Profile name | Stable CLI profile name, for example `tls` or `oidc`. |
| Fixture list | Exact generated paths, grouped by role. |
| Positive cases | The happy-path behavior downstream tests can assert. |
| Negative cases | The rejection paths or malformed shapes downstream tests can assert. |
| Receipts | Machine-readable generation, audit, or materialization records. |
| Claim boundary | Explicit statement of what the pack does not prove. |
| Proof command | `cargo xtask bundle-proof --profile <name> --out <path>` or an equivalent command-backed proof. |
| Task-first how-to | User guide that starts from the test problem, not implementation internals. |
| Release-evidence row | Patch, minor, or scheduled lane that carries the pack proof. |
| Scanner-safety rule | Whether generated bytes are scanner-safe and whether users should commit or regenerate them. |

New contract packs must follow this lifecycle:

```text
proposal or design note
  -> accepted spec
    -> implementation
      -> proof command
        -> how-to guide
          -> claim-ledger entry
            -> release-evidence row
```

The proof command must exist before the pack is described as stable in README,
release notes, or the public claim ledger.

## Non-goals

This spec does not add a new contract pack.

This spec does not expand the TLS pack with mTLS, revocation, OCSP, certificate
transparency, browser trust-store behavior, or variable hostname matrices.

This spec does not require every fixture family to become a contract pack. A
single fixture API can remain a normal crate feature if it does not need a
profile, receipt, and task-first how-to.

This spec does not make full pack proofs mandatory for every patch PR. Release
lane specs decide when full pack proofs run.

## Required Evidence

The current reference proof is TLS:

```bash
cargo xtask bundle-proof --profile tls --out target/release-evidence/tls
```

The OIDC/JWKS pack follows the same proof shape:

```bash
cargo xtask bundle-proof --profile oidc --out target/release-evidence/oidc
```

Docs-only changes to this spec should run:

```bash
cargo xtask docs-sync --check
cargo xtask typos
git diff --check
```

When a PR changes pack behavior, it should also run the pack-specific proof
command and update the claim ledger or user docs if the boundary changes.

## Acceptance

This spec is accepted when:

- it defines the required fields of a stable contract pack;
- it defines the lifecycle from proposal/design to release evidence;
- it identifies TLS as the reference implementation;
- it keeps pack stability tied to proof commands and user docs.

This spec is implemented when:

- every stable contract pack has a claim-ledger entry;
- every stable contract pack has a task-first how-to;
- every stable contract pack has a bundle-proof or equivalent proof command;
- release evidence maps minor-release pack claims to their proof artifacts.

## Acceptance Examples

### TLS Reference Pack

Profile:

```bash
uselesskey bundle --profile tls --out target/tls-fixtures
```

Proof:

```bash
cargo xtask bundle-proof --profile tls --out target/release-evidence/tls
```

User doc:

```text
docs/how-to/test-tls-chain-validation.md
```

Generated proof artifacts:

```text
target/release-evidence/tls/tls-contract-pack-proof.json
target/release-evidence/tls/tls-contract-pack-proof.md
```

Required fixture classes:

- valid leaf;
- valid chain;
- expired leaf;
- not-yet-valid leaf;
- wrong-hostname leaf;
- untrusted-root leaf;
- materialization receipt;
- audit-surface receipt;
- evidence markdown.

Boundary:

```text
The TLS contract pack proves the documented generated fixtures, receipts, and
negative validation paths. It does not prove mTLS, revocation, CT, browser
trust-store behavior, production CA custody, or downstream verifier correctness.
```

### OIDC/JWKS Reference Pack

Profile:

```bash
uselesskey bundle --profile oidc --out target/oidc-fixtures
```

Proof:

```bash
cargo xtask bundle-proof --profile oidc --out target/release-evidence/oidc
```

User doc:

```text
docs/how-to/test-oidc-jwks-validation.md
```

Boundary:

```text
The OIDC/JWKS pack proves deterministic key-set and token-shape fixtures plus
documented negatives. It does not prove production signing-key custody, full
OpenID provider behavior, or downstream validator correctness.
```

## Test Mapping

Contract-pack tests are split by lane:

- CLI profile generation and verification tests cover command behavior.
- Bundle-proof commands cover end-to-end generation, verification, inspection,
  export, and receipt production.
- `cargo xtask no-blob` covers committed secret-shaped fixture policy.
- Task-first how-to docs cover user workflow expectations.
- Release evidence carries the proof artifacts for shipped claims.

TLS reference evidence:

```bash
cargo xtask bundle-proof --profile tls --out target/release-evidence/tls
cargo xtask no-blob
```

OIDC/JWKS reference evidence:

```bash
cargo xtask bundle-proof --profile oidc --out target/release-evidence/oidc
cargo xtask no-blob
```

## Implementation Mapping

Contract-pack requirements are owned by:

- `crates/uselesskey-cli` for `bundle`, `verify-bundle`, `inspect-bundle`, and
  `export` surfaces;
- owner crates such as `uselesskey-x509`, `uselesskey-jwk`, and
  `uselesskey-token` for generated fixture behavior;
- adapter crates when a pack exposes typed downstream helpers;
- `xtask` for `bundle-proof` and release-evidence wiring;
- `docs/how-to/` for task-first user workflows;
- `policy/claim-ledger.toml` for public claim mapping;
- `docs/status/PUBLIC_CLAIMS.md` for reader-facing claim status.

## CI Proof

Docs-only changes to this spec should run:

```bash
cargo xtask docs-sync --check
cargo xtask typos
git diff --check
```

Pack behavior changes should add the relevant proof:

```bash
cargo xtask bundle-proof --profile tls --out target/release-evidence/tls
cargo xtask bundle-proof --profile oidc --out target/release-evidence/oidc
```

Release evidence should include stable pack proofs in the minor lane. Patch
release evidence may omit full pack proofs unless the patch touches pack
behavior or claim text.

## Metrics / Promotion Rule

A new pack can move from proposed to stable only when it has:

- an accepted spec or design note;
- implementation and tests;
- a proof command;
- a task-first how-to guide;
- a claim-ledger entry;
- release-evidence mapping;
- an explicit boundary.

This spec can move to `implemented` when `spec-check` can validate that stable
contract-pack claims have docs, proof commands, and claim-ledger entries.
